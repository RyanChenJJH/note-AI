# 03 · 数据源与解析 ⭐

> 本文件是整个工具的地基。两种数据源的格式都是**实测**得到的（2026-06，本机），上游可能演进，解析需容错。

---

## A. Claude Code

### A.1 位置

```
%USERPROFILE%\.claude\projects\<编码后的项目路径>\<会话uuid>.jsonl
```

- 一个会话 = 一个 `.jsonl` 文件，文件名是会话 UUID。
- 目录名是把项目绝对路径里的 `:` `\` `_` 都替换成 `-`（**有歧义、不可逆**）。
  例：`E:\Work2\AI_Work\caude_code_project\03_note_ai`
  → `E--Work2-AI-Work-caude-code-project-03-note-ai`
- ⚠️ **不要靠目录名反推项目路径**。每条记录内部都有 `cwd` 字段，用它即可。

### A.2 记录结构

每行一个 JSON 对象，按 `type` 区分。实测见到的 `type`：
`permission-mode`、`file-history-snapshot`、`user`、`assistant`、`attachment`（可能还有 `system` 等）。

公共字段（视类型而定）：`type, uuid, parentUuid, timestamp, sessionId, cwd, gitBranch, version, message, isSidechain, userType`。

**人类提问**（`type: "user"`）：
```jsonc
{
  "type": "user",
  "message": { "role": "user", "content": "你可以做什么工作" },  // content 是字符串
  "timestamp": "2026-...Z", "cwd": "C:\\Users\\...", "sessionId": "...", ...
}
```

**AI 回复**（`type: "assistant"`）：`message.content` 是**块数组**：
```jsonc
{
  "type": "assistant",
  "message": {
    "role": "assistant",
    "content": [
      { "type": "thinking", "thinking": "..." },     // 思考（默认不导出）
      { "type": "text", "text": "可见的回复正文" },    // ← 要的
      { "type": "tool_use", "name": "Bash", "input": {...} }  // 工具调用（丢弃）
    ]
  }
}
```

**工具结果**：会以 `type: "user"` 回流，但 `message.content` 是数组且含 `{type:"tool_result",...}` —— **这不是人类提问**，要排除。

### A.3 过滤规则（Claude）

| 记录 | 处理 |
|------|------|
| `type:user` 且 `content` 是**字符串** | ✅ 人类提问 |
| `type:user` 且 `content` 是数组且有 `{type:"text"}` 且无 `tool_result` | ✅ 人类提问（取 text） |
| `type:user` 且含 `tool_result` | ❌ 丢弃（工具结果） |
| `type:assistant` 的 `text` 块 | ✅ AI 回复正文 |
| `type:assistant` 的 `thinking` 块 | ⚙️ 默认丢弃，可由"含思考过程"开关保留 |
| `type:assistant` 的 `tool_use` 块 | ❌ 丢弃 |
| `isSidechain: true` 的记录 | ❌ 丢弃（子任务/Agent 旁路） |
| `permission-mode` / `file-history-snapshot` / `attachment` / `system` | ❌ 丢弃 |

## B. Codex CLI

### B.1 位置

```
%USERPROFILE%\.codex\sessions\YYYY\MM\DD\rollout-<ISO时间>-<uuid>.jsonl
```

- 另有 `%USERPROFILE%\.codex\session_index.jsonl`（会话索引，可用于快速列清单，需另行确认字段）。

### B.2 记录结构

每行一个 JSON 对象，外层都有 `{timestamp, type, payload}`。实测 `type`：
`session_meta`、`event_msg`、`response_item`、`turn_context`。

**首行 `session_meta`**：含会话级元信息（cwd、起始时间等），用于标题前的项目/时间。

**消息**（`type: "response_item"` 且 `payload.type: "message"`）：
```jsonc
{
  "type": "response_item",
  "payload": {
    "type": "message",
    "role": "user",            // user | assistant | developer
    "content": [
      { "type": "input_text",  "text": "你好" }        // user 用 input_text
      // assistant 用 output_text；也可能是 text
    ]
  },
  "timestamp": "2026-...Z"
}
```

正文 = `payload.content[].text` 拼接。

### B.3 过滤规则（Codex）

| 记录 | 处理 |
|------|------|
| `response_item` + `message` + `role:user`，且正文**不是**系统包裹 | ✅ 人类提问 |
| `response_item` + `message` + `role:assistant` | ✅ AI 回复 |
| `role:developer` | ❌ 丢弃（系统指令） |
| user 正文以 `<environment_context>` / `<permissions instructions>` 等尖括号包裹开头 | ❌ 丢弃（注入的上下文，非真人输入） |
| `payload.type` 为 `function_call` / `function_call_output` / `reasoning` 等 | ❌ 丢弃（工具/思考；reasoning 可选保留） |
| `event_msg`（如 `task_started`）/ `turn_context` / `session_meta` | ❌ 丢弃（meta） |

> Codex 一个会话开头通常是：`developer`(权限说明) → `user`(environment_context) → 才到真正的第一句人话。**第一条真人 user** 才用来当标题。

## C. 统一的"分轮"算法

两个源解析后都归一化成 `Round[]`。一轮 = **一次真人提问 + 其后 AI 产生的全部可见文字回复**（中间的工具循环都并入这一轮的回复）。

```text
rounds = []
current = null
for rec in 过滤后按时间排序的记录:
    if rec 是“人类提问”:
        if current: rounds.push(current)
        current = { index: rounds.length+1,
                    question: rec.text,
                    answerParts: [] }
    elif rec 是“AI 文字回复” and current:
        current.answerParts.push(rec.text)
if current: rounds.push(current)

# answer = answerParts 用换行/分隔拼接
# 若某轮没有 answer（例如最后一句没回复），answer 置空，块标记可禁用勾选
```

要点：
- 先**过滤**再**按 `timestamp` 排序**，避免乱序。
- Claude 里同一轮可能有多条 `assistant` 记录（工具循环），全部并入 `answerParts`。
- 标题 = `rounds[0].question` 截断（建议 40 字内）。
- 轮数 `turnCount = rounds.length`。

## D. 会话级元信息提取

| 字段 | Claude 来源 | Codex 来源 |
|------|-------------|------------|
| `sessionId` | 文件名 / 记录 `sessionId` | **文件名里的 uuid**（⚠️ 不可用 `session_meta.id`：那是线程 id，resume/续接时被多个 rollout 文件复用会撞车，见 docs/16 BUG-001） |
| `startedAt` | 第一条记录 `timestamp` | `session_meta.timestamp` / 首条 |
| `project`(cwd) | 记录 `cwd` | `session_meta` 里的 cwd |
| `title` | 首条真人提问 | 首条真人提问 |
| `day` | `startedAt` 取 `YYYY-MM-DD` | 同左 |

## E. 容错与边界

- 遇到**未知 type / payload.type**：跳过，不报错。
- 单行 JSON 解析失败：跳过该行并记 warning，继续。
- 空会话（无真人提问）：列表中可隐藏或标记"无内容"。
- 超长正文：解析不截断，**仅在列表标题处截断**显示。
- 编码统一按 **UTF-8** 读取。
- 上游格式若变动：优先在本文件更新"过滤规则"，parser 模块对照实现。
